# Talent Assessment & Performance: Requiem

**Version**: 1.0.0  
**Last Updated**: 2026-03-02

## 1. Interview Rubric: Technical Invariants
*We don't hire for "Years of Experience"; we hire for "Mental Models."*

| Competency | Score (1-5) | Signal to Look For |
|------------|-------------|---------------------|
| **Determinism Mindset**| | Can they explain why `Math.random()` or `new Date()` are problematic in a distributed system? |
| **Security First** | | Do they automatically think about "Deny-by-Default"? Do they check for bypasses? |
| **System Empathy** | | Do they understand the performance cost of their abstractions? (e.g., C++ vs. Python overhead). |
| **Rigorous Execution**| | Is their code clean, tested, and free of "TODO" markers? |

## 2. Equity & Compensation Guidelines
*Aligned with "Antigravity"—rewarding structural contribution.*

- **Founding Team**: 1-3% Equity. Market salary + 20% "Integrity Bonus" based on OKRs.
- **Lead / Principal**: 0.5-1% Equity. Top 10% market salary.
- **Senior**: 0.1-0.3% Equity. Market competitive.

## 3. Performance Review Template
*The "Entropy Pass"*

1. **Structural Coherence**: Did your contributions simplify the system or add complexity?
2. **Determinism Score**: How many of your features required bug-fixes due to non-deterministic behavior?
3. **Guardrail Contribution**: Did you add CI tests or lint rules to prevent regressions? (Antigravity mode requirement).
4. **Peer Velocity**: Did you make those around you faster by providing clear docs or interfaces?

## 4. Onboarding Checklist (First 30 Days)
- [ ] **Day 1**: Run `reach run` and `verify` locally.
- [ ] **Day 7**: Successfully commit a new policy rule and see it pass CI.
- [ ] **Day 14**: Draft a "Mini-RFP" for a new feature, emphasizing security invariants.
- [ ] **Day 30**: Present one improvement to the "Deterministic Verification" suite.
