# Security Policy

> **Honesty over Theatre.**

Requiem is designed for high-security AI execution, but we are honest about our current limitations. We maintain a live audit of implemented vs. stubbed security controls in [docs/THEATRE_AUDIT.md](./docs/THEATRE_AUDIT.md).

---

## Reporting a Vulnerability

If you discover a security vulnerability, please do NOT open a public issue. We appreciate reports that help us keep our users safe.

**Disclosure Process:**
1.  **Report**: Email your findings to [security@ready-layer.com](mailto:security@ready-layer.com).
2.  **Acknowledgment**: We will acknowledge your report within 24 hours.
3.  **Triage**: We will provide an initial assessment and estimated fix timeline within 72 hours.
4.  **Fix**: We will work with you to validate the fix before public disclosure.

---

## Security Primitives

Requiem uses the following primitives to ensure execution integrity:
- **Hashing**: BLAKE3 (with domain separation).
- **Isolation**: OS-level sandboxing (Job Objects on Windows, rlimits on Linux).
- **Audit**: Merkle-linked event logs.

### Known Limitations
- Production JWT validation is currently a stub in the open-source version.
- Seccomp-BPF syscall filtering is in progress for Linux builds.
